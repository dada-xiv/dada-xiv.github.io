Git Initial Setup

After installing Git, it is important to configure the Git environment properly. This configuration process needs to be done only once on a computer, and the settings will be maintained even if Git is upgraded. You can use a tool called 'git config' to check and modify these settings. Git operates based on these configurations. During the setup, three files are involved:
  • /etc/gitconfig: This file contains settings that apply to all users and repositories on the system. It can be read and modified using the 'git config --system' option. Modifying this file requires system administrator privileges as it is a system-wide configuration file.
  • ~/.gitconfig, ~/.config/git/config: These files contain settings that are specific to a particular user, also known as the current user. You can read and modify these files using the 'git config --global' option. These settings apply to all repositories associated with the specific user.
  • .git/config: This file contains settings that are specific to a particular repository. It is located in the .git/ directory of the repository. By default, the '--local' option is applied, which means the settings in this file are used. To use this file explicitly, you can specify the '--local' option after moving to the respective repository directory.
Each setting takes precedence in reverse order, meaning that the repository-specific settings override the user-specific settings, and the user-specific settings override the system-wide settings.

Set User Information

The first step after installing Git is to configure your username and email address. Git utilizes this information for every commit, and once committed, it cannot be changed.

To set up your username and email address, execute the following commands: 
$ git config --global user.name 'yourname'
$ git config --global user.email 'your@email.com'
Replace 'yourname' with your desired username and 'youremail@example.com' with your actual email address.

Check Settings

To verify your Git settings, you can use the following command: 
$ git config --list
Executing this command will display all the configurations you have set up, allowing you to review them immediately.

Additionally, if you want to retrieve the value of a specific configuration key, you can use the following command: 
$ git config user.name
This command will display the value of the 'user.name' configuration key, which represents your username. If you change your username into a specific name, use: 
$ git config user.name {yourname}

Set the Default Branch Name of Newly Created Repositories to 'main' Instead of 'master'

In June 2020, there was a movement in the IT industry to remove terms that were perceived to have racial or master-slave connotations, such as whitelist/blacklist and master/slave. This movement also impacted Git, which previously used 'master' as the default branch name.

In order to align with these changes, Git made it possible for users to specify their own default branch name. Following suit, GitHub also decided to change the default branch name from 'master' to 'main', and this change was implemented on October 1, 2020.

To set the default branch name to 'main', you can add the following configuration to either the '/etc/gitconfig' or '~/.gitconfig' file: 
[init]
	defaultBranch = main
By adding this configuration, any newly created repositories will have 'main' as the default branch instead of 'master'.

Editor Settings

By default, Git utilizes the system's default editor. However, you can configure Git to use a specific editor of your choice. For example, if you want to set 64-bit Notepad as the default editor on a 64-bit Windows system, follow these steps: 
$ git config --global core.editor "'C:/Program Files/Notepad++/notepad++.exe' -multiInst -nosession"
With this command, you are configuring Git to use the 64-bit Notepad++ editor located at the specified path.

To open a specific file, such as 'README.md', using the editor that has been set as the default, you can use the following command: 
$git config --global --edit README.md
This command will open the 'README.md' file in the editor specified in the global Git configuration.

Create Repository: Initialize Directory

To initialize a specific directory as a Git repository, navigate to that directory and perform the following steps: 
$ git init
Alternatively, if you want to create a directory named "apple" and initialize the repository within it, use the command: 
$ git init apple
This will create the "apple" directory and initialize the repository inside it.

To remove the repository and undo the configuration: 
$ rm -r .git
This command will delete the corresponding ".git" folder, effectively uninitializing the repository.

Copy from an Existing Repository

To clone an existing repository, use the 'git clone {url}' command. For example: 
$ git clone https://github.com/libgit2/libgit2
This command will clone the repository named "libgit2" along with the ".git" directory.

If you want to create a repository named "mylibgit" while cloning, use the command: 
$ git clone https://github.com/libgit2/libgit2 mylibgit
This will clone the repository and create a new directory named "mylibgit" for the cloned repository.

In addition to using the "https://" protocol, you can also utilize other protocols such as "git://" or "user@server:path/to/repo.git" using SSH.

Exclude from Version Control

To exclude specific files, directories, and extensions from being tracked by Git, you can create a '.gitignore' file. This file contains patterns that Git uses to determine what should be excluded. Here's an example of a '.gitignore' file: 
$ cat .gitignore
*.[oa]
*~

# Exclude files with the extension .a, except for lib.a
!lib.a

# Exclude TODO file in the current directory, but not in subdirectories like subdir/TODO
/TODO

# Exclude all files in the build/ directory
build/

# Exclude doc/notes.txt file, but not doc/server/arch.txt file
doc/*.txt

# Exclude all .pdf files under the doc directory, including subdirectories
doc/**/*.pdf
In this example, the first line instructs Git to ignore files with the extensions '.o' or '.a'. The second line excludes files ending with '~'. The notation '[abc]' specifies any one of the characters inside the brackets.

Directories are represented by using a forward slash (/) at the end. The patterns are relative to the location of the '.gitignore' file.

By defining exclusion patterns in the '.gitignore' file, you can prevent Git from tracking certain files or directories in your repository.

Working with Git : Record Changes

In Git, when you want to capture a specific state of your project, you begin by making changes and then commit a snapshot of those changes to the repository. Each file in the working directory can be classified as either tracked or untracked.

Tracked files encompass two categories: the files that were included in the last snapshot and newly staged files. These tracked files can be in one of three states: unmodified, modified, or staged. In essence, a tracked file is a file that Git is aware of.

On the other hand, untracked files refer to all the files in the working directory that are not part of the last snapshot and have not been staged. When you initially clone a repository, all files are considered tracked and unmodified since Git has checked them out, and you haven't made any edits yet.

As you make edits to a file, Git recognizes it as modified because the file has changed since the last commit. Throughout your work, you have the option to selectively stage these modified files and subsequently commit all the staged changes.

Thus, the cycle of Git follows the sequence of unmodified, modified (staging), and committed. This cycle repeats as you continue working on your project, capturing changes in an organized manner. To check the current status of your repository, you can use the following commands: 
$ git status
$ git status -s
The first command provides a detailed overview of the current status, while the second command gives a more concise summary.

Staging and Commit

To create a new commit, you need to add the changes you want to include in the commit to the staging area. Here's an example of creating the first commit by adding a '# git-howto' line to the README.md file: 
$ echo "# git-howto" >> README.md
$ git add README.md
$ git commit -m "first commit"
You can add individual files to the staging area using the 'git add' command, like 'git add hello.txt'. To add all modified files from the current repository to the staging area at once, you can use 
$ git add .
The following command adds all modified files, including untracked files, to the staging area: 
$ git add -A
Before committing your changes, it's important to check the files in the staging area to ensure that you have included all the desired modifications. Files in the unstaged state are not committed. You can use the 'git status' command to review the staging status of your files. Once you are satisfied with the changes in the staging area, you can commit them using the 'git commit' command: 
$ git commit
$ git commit -m 'message1'
The second command implies to commit with the message 'message1'.

To add all modified and untracked files in your working directory to the staging area and commit them, you can use the following commands: 
$ git commit -am 'message2'
$ git commit -a -m 'message2'
These two lines are identical in their effect.

To edit the message of the latest commit, you can use the following command: 
$ git commit --amend
This command opens the default text editor with the previous commit message, allowing you to make changes. Note that amending a commit replaces the previous commit with a new one, so use it with caution, especially if the commit has already been pushed to a remote repository.

Examine the commit history

To examine the commit history and track changes in Git, you can use the following commands. To view the commit history: 
$ git log
This command displays a detailed log of all commits in the repository, including commit messages, authors, dates, and unique commit hashes.

To display a single-line summary of the commit history: 
$ git log --oneline
This command provides a concise overview of the commit history, showing only the commit hash and the commit message.

To show the commit status of each branch: 
$ git log --branches
This command displays the commit history specific to each branch in the repository.

To visualize the commit history as a graph: 
$ git log --graph
This command represents the commit history as a graph, showing branches, merges, and commit relationships.

To combine multiple options for a comprehensive view of the commit history: 
$ git log --oneline --branches --graph
This command combines the options to display a condensed, branch-specific commit history with a graphical representation.

To see a summarized view of changes in the commit history: 
$ git log --stat
This command provides a detailed overview of changes in each commit, including the number of modified files and the number of lines added or removed.

To check the differences between the version in the working tree and the latest committed version/repository: 
$ git diff
This command shows the differences between the current state of files in the working tree and the last committed version in the repository.

It's important to note that the 'git diff' command shows unstaged changes, while the 'git status' command provides information about both staged and unstaged changes. Use 'git diff' to examine specific line changes or patches, and 'git status' for a high-level overview of the repository's current state.

이제는 https:// 프로토콜로도 Git 저장소를 사용하는 데 부족함이 없다. 간단히 사용자이름과 암호로 인증만 하면 된다. 공개 프로젝트를 Clone 하는 데는 인증도 필요 없다.

SSH 리모트를 쓰려면 공개키를 설정해야 한다. (아직 공개키가 없으면 SSH 공개키 만들기를 참고) 아래 Windows의 오른쪽 꼭대기에 있는 계정 설정 링크를 클릭하자. 그 왼쪽 메뉴에서 “SSH keys” 를 선택한다. 여기서 “Add an SSH key” 버튼을 클릭한다. 키 이름을 적당히 입력하고 ~/.ssh/id_rsa.pub 파일의 내용을 입력 칸에 복사해 넣는다. 그리고 “Add key” 버튼을 클릭한다.

이제 GitHub의 저장소에서 'Clone with SSH'에 있는 주소를 복사한 뒤
$ git remote add origin git@github.com:계정/connect-ssh.git
으로 하면 ssh로 연결된다.

Git에서 먼저 ssh 공개키를 생성하고 GitHub에 등록하려면 다음과 같이 한다: 
$ cd ~
$ cd ssh-keygen
$ cd ~/.ssh
$ cat id_rsa.pub
하면 출력되는 문자열 전체를 복사한다. GitHub의 setting에서 [SSH and GPG keys] 항목에 복사한 퍼블릭키를 등록한다. (SSH 퍼블릭 키는 여러 개를 등록할 수 있다.)

Get remote branch information

Here's the revised version with improved grammar and expression: While 'git pull' fetches commits from the remote repository to update the local repository, 'git fetch' only retrieves information about any changes in the remote branch. After reviewing the changes, you can browse the source and decide whether to commit them to the local repository.

To fetch from the remote repository, navigate to a directory in your local repository and run the command: 
$ git fetch
This command retrieves the changes from the remote repository, but doesn't apply them locally. If you use the 'git log' command, you will see that only the 'main' branch is visible, while the remote repository's 'origin/main' is not.

To check the differences between your local current branch and the remote 'origin/main', use the command: 
$ git status
This will inform you of the changes and prompt you to reflect them using 'git pull'. When you run 'git fetch', the latest commit information of the remote branch is stored in a branch called 'FETCH_HEAD'. To view the contents of the latest fetched commit, use the following commands: 
$ git checkout FETCH_HEAD
$ git log
After reviewing the contents, you can decide whether to apply the latest commit from the remote branch to your local repository. To switch back to the 'main' branch, use: 
$ git checkout main
To merge the latest commits that have not been reflected locally after fetching, use the command: 
$ git merge FETCH_HEAD
If you want to merge in one step, you can directly merge the commits from the remote main branch into the local branch: 
$ git merge origin/main
Similarly, you can merge commits from the remote 'apple' branch to the local branch: 
$ git merge origin/apple
These commands allow you to merge changes from specific remote branches into your local repository.

Change the name of repository

To change the name of a GitHub repository while maintaining its connection to local computer, first go to the local repository and check the current remote repository URL(s): 
$ git remote -v
This will display 
origin  https://github.com/{user-id}/{repository-name}.git (fetch)
origin  https://github.com/{user-id}/{repository-name}.git (push)
Now to rename the repository on GitHub, Go to the GitHub website and navigate to the repository's page. Then click on the "Settings" tab and change the "Repository name" to the new name. Then update the remote URL for the local repository: 
$ git remote set-url origin {new_repository_url}
After this, verify that the remote URL has been updated by running $ git remote -v command again. You should see the new repository URL.

협업을 시작할 때 공동 작업자는 자신의 작업을 시작하기 전에 git clone을 통해 원격 저장소를 복제한 후 사용한다:
$ git clone 원격 저장소 주소

협업 과정에서는 여러 사람이 함께 문서를 수정/push하기 때문에 반드시 원격 저장소의 최신 커밋을 pull한 다음 자신의 커밋을 push해야 한다.
$ git pull
하여 작업 후
$ git push -u origin main
하여 원격에 반영한다. 원격 저장소의 최신 커밋이 반영되지 않은 상태에서 로컬에서 수정 후 커밋을 시도하면 ![rejected]로 시작하는 오류 메세지가 뜬다.

GitHub(+협업)에서 브랜치 사용하기

로컬 저장소에서
$ git remote -v
를 실행하여 연결 상태를 확인한 뒤
$ git pull
에서 최신 커밋을 받아온다.
$ git checkout -b f (f라는 새로운 브랜치를 만들고 동시에 체크아웃)
새로운 브랜치 f에서 작업을 한 뒤 커밋한 다음
$ git push origin f (원격 저장소 origin에 f 브랜치를 push)
github에서 확인하면 방금 push한 브랜치가 나타난다.

이렇게 push한 브랜치는 pull request를 통해 병합해야 원격 저장소(github)에 반영된다. 브랜치 설정에서 [New pull request] 등을 클릭하여 나타나는 'Open a pull request' 창에 풀 리퀘스트 메세지를 작성한 뒤 [Create pull request]를 누르면 협업 중인 저장소에 풀 리퀘스트가 전송된다.

협업 중인 원격 저장소에 등록된 풀 리퀘스트는 공동 작업자 중 누구나 살펴보고 병합할 수 있다. 저장소 파일 목록 위의 [Pull request] 탭을 누르면 등록된 풀 리퀘스트 목록이 나타난다. 방금 등록한 pull request를 누르고 내용을 확인한 뒤 문제가 없으면 [Merge pull request]를 눌러 병합한다. 이 창에서 풀 리퀘스트를 남긴 사람과 메시지를 주고받을 수 있다. [Confirm merge]를 누르면 브랜치 병합이 끝난다.

브랜치가 병합되면 해당 브랜치에 있던 파일이 main에도 나타날 것이다. 브랜치 전체 상태를 알고 싶다면 파일 목록 위의 '2 brinches'를 클릭한다. 브랜치가 병합된 상태라면 'merged'라는 표시가 붙는다. 그리고 공동 작업자 중 누가 브랜치를 병합했는지도 알 수 있다.

github로 협업할 때는 작업자들마다 브랜치를 만들어 진행하고, 작업 중간중간 플 리퀘스트를 보내서 master 브랜치에 병합한다. 그러므로 다른 작업자들의 현재까지의 변경 내용을 바로 반영하기 위해 항상 pull 부터 한 다음 자신의 작업을 진행하는 것이 좋다.
